Act as a principal product manager and principal software engineer: optimize for user value, correctness, maintainability, clear tradeoffs, and small reviewable changes.

## Repo Shape

- Nx monorepo for a Kubernetes-native, GitOps-oriented platform.
- Package manager: npm workspaces. Prefix Nx commands with `npm exec nx`, for example `npm exec nx run <project>:build`.
- `apps/`: deployable apps and e2e projects.
- `packages/`: reusable domain, data, UI, model, and test libraries.
- `deploy/`: Helm, Kustomize, and Argo CD manifests when deployment config is added.
- `infra/`: OpenTofu or Crossplane code when infrastructure ownership is added.
- Platform defaults: Azure DevOps CI, Argo CD CD, Helm, Kustomize overlays, External Secrets Operator, Argo Rollouts, Prometheus/Grafana, Loki, OpenTelemetry. Service mesh is optional only when its complexity is justified.

## Agent Workflow

1. Inspect existing structure, conventions, and affected Nx projects before changing code.
2. Preserve Nx project boundaries and dependency direction.
3. Make the smallest safe change; avoid broad rewrites and new frameworks unless the tradeoff is explicit.
4. Add or update tests for behavior changes.
5. Update deploy, infra, docs, or observability when runtime behavior changes.
6. Run the CI-equivalent Nx validation locally before push so lint, tests, builds, and e2e are already passing and Nx cache is warm.
7. After push, babysit CI and GitHub checks until they pass or the failure is understood; do not call work done while checks are still pending or failing.

## Common Commands

- Install: `npm install`
- Explore graph: `npm exec nx graph`
- Affected lint: `npm exec nx affected -t lint`
- Affected test: `npm exec nx affected -t test`
- Affected build: `npm exec nx affected -t build`
- Affected e2e: `npm exec nx affected -t e2e`
- Project target: `npm exec nx run <project>:<target>`
- Local stack: prefer `npm run dev` once available.

## Architecture Rules

- Keep applications, reusable libraries, infrastructure, and deployment config separate.
- Keep business logic out of `deploy/` and `infra/`; keep deployment concerns out of app code.
- Prefer shared libraries over repeated logic once reuse is real and ownership is clear.
- Avoid circular dependencies and cross-service imports that bypass APIs, events, or contracts.
- Recommended flow: `apps -> feature libs -> domain libs -> shared utility libs`; services follow the same inward direction.
- Shared libraries need a clear purpose. Avoid generic junk-drawer modules such as `common`, `utils`, or `shared-stuff`.
- Introduce a new app, library, microfrontend, or microservice only when it improves clarity, ownership, testability, build performance, runtime scalability, deployment safety, reuse, security, maintainability, or developer velocity.

## Deployment And Infrastructure

- CI validates; it does not deploy production directly.
- CD is GitOps through Argo CD watching declarative manifests under `deploy/`.
- Use Helm for deployable app packaging: Deployment, Service, Ingress/Gateway, ConfigMaps, ServiceAccount, HPA, PDB, resources, probes, and optional Argo Rollouts.
- Use Kustomize base plus overlays for `dev`, `qa`, `staging`, and `prod`; overlays own replica counts, resources, domains, config, feature flags, rollout strategy, and ingress/gateway differences.
- Do not duplicate full manifests across environments unless unavoidable.
- Use OpenTofu for infrastructure managed outside the cluster and Crossplane for Kubernetes-native provisioning. Do not let both own the same resource.
- Deploy pipeline tests must respect blue/green flow: validate the current blue environment first, validate the green candidate before traffic moves, and flip traffic only after the required tests pass. Failed green deployments stay unpromoted and must be cleaned up or rolled back.

## Security And Secrets

- Never commit secrets, tokens, API keys, passwords, or sensitive customer data.
- Use External Secrets Operator: external store holds values, `ExternalSecret` manifests live in Git, Kubernetes Secrets are generated by the operator.
- Containers should run as non-root where possible; RBAC should follow least privilege.
- Public endpoints must be explicit.
- Dependency and container image scanning belong in CI.
- Authentication and authorization behavior needs test coverage.

## Observability And Reliability

- Every deployable service needs health and readiness endpoints, structured logs, useful metrics, and clear dependency failure visibility.
- Logs should include correlation/request ID, safe user or tenant context, operation name, duration, success/failure state, and useful error details.
- Never log passwords, tokens, API keys, secrets, full PII payloads, or sensitive customer data.
- Track request count, duration, error rate, dependency health, queue depth, background jobs, and business-critical workflows.
- Use OpenTelemetry tracing where it clarifies cross-service, dependency, queue, or external-system flow.
- Production services need resource requests/limits, graceful shutdown, probes, rollback through Git history, and PDBs for critical workloads.
- Use progressive delivery for production changes when risk justifies it.

## Engineering Budgets

- Local stack starts in under 1 minute from a fresh checkout with one clear command.
- Local ports must be deterministic per working directory or managed so multiple worktrees can run together.
- Affected CI path for build, test, lint, e2e, and deploy prep should stay under 10 minutes.
- Split long-running operations into explicit Nx targets instead of hiding them inside broad builds or library internals. For example, Hasura metadata work should have separate targets such as `metadata-generate` and `replace-metadata` so agents and CI can run, cache, retry, and parallelize only the needed step to achieve budgets.
- File target: under 800 lines; prefer under 400.
- Function target: under 50 lines; prefer under 30. Keep nesting at 3 levels or less and cyclomatic complexity under 10.
- Frontend bundle target: initial JS under 250 KB gzipped; route chunks under 150 KB gzipped.
- API p95 targets: simple reads under 300 ms, complex reads under 800 ms, writes under 1000 ms, background job acknowledgement under 500 ms.
- Tests: unit file under 5 seconds, project unit tests under 60 seconds, critical e2e suite under 5 minutes, flaky test tolerance is zero.

## Resource Consciousness

- Root issue: parallelism is harmful when every worker or worktree starts its own full Docker Compose/dev stack. Scale test clients, not local stacks.
- Allow one active `npm run dev` stack per repository path. Dev/e2e setup must detect and reuse prior instances it created, handle stale locks, and clean up processes, containers, and volumes on exit or interruption.
- Build cacheable artifacts first with Nx, then start Docker images/compose and the local stack. Do not hide expensive build, metadata, or generation work inside stack startup.
- Parallel e2e workers should point to the same prepared local instance unless a test explicitly requires full stack isolation.

## Testing

- Match test level to risk: unit for business logic, integration for service boundaries, contract tests for service communication, e2e for critical user flows, smoke tests after deployment.
- Do not use e2e tests for every small behavior.
- The stack must bootstrap one management superadmin organization/user. Additional seeding is optional, and the environment must function without seed data.
- Every test must create its own user, organization, and resource namespace; tests must not step on each other's data and must clean up what they create.
- Test-created object names must include a date stamp and unique suffix so interrupted runs are traceable. Automated cleanup should remove test objects with date stamps older than 2 days.
- Migrations must be versioned, production-aware, and backward compatible when possible.
- Destructive migrations require explicit review; large data migrations must not run in request paths.
- Use expand-and-contract patterns when rolling out schema changes.

## Pull Requests And Done Criteria

- Keep PRs small, focused, and easy to review.
- PR notes should cover what changed, why, testing, deployment impact, rollback notes, and screenshots or recordings for UI changes.
- Before push, all CI-equivalent checks must pass locally through Nx so CI can reuse cache and finish faster.
- After push, monitor CI and GitHub checks to completion; fix failures or clearly report the blocker before marking the work done.
- Done means code is implemented, relevant tests are updated, lint/test/build pass, runtime config is updated when needed, observability and secrets are considered, and rollback impact is understood for production changes, functional testing is completed ideally via Chrome MCP Server like a human would test.
